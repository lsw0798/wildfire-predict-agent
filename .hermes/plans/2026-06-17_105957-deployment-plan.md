# 산불 의사결정 지원 웹서비스 배포 계획

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** 현재 로컬 검증이 끝난 산불 모니터링/분석 웹서비스를 외부 접속 가능한 형태로 안전하게 배포한다.

**Architecture:** 프론트엔드(Next.js)와 백엔드(FastAPI)를 분리 배포하되, 초기 배포는 가장 단순한 2서비스 구조로 진행한다. 프론트는 정적/서버 렌더링이 가능한 Node 런타임, 백엔드는 Python API 런타임으로 배포하고, 환경변수·CORS·헬스체크·도메인 연결을 먼저 안정화한 뒤 최종 smoke test를 수행한다.

**Tech Stack:** Next.js 15, React 19, FastAPI, Uvicorn, Playwright, GitHub Actions, 환경변수 기반 설정

---

## 1. 배포 목표

이번 배포의 1차 목표는 아래 4가지를 만족하는 것이다.

1. 외부 사용자가 `/` 모니터링 화면과 `/analyze` 화면에 접속 가능
2. 프론트에서 백엔드 `/api/monitoring`, `/api/analyze`, `/health`를 정상 호출 가능
3. 현재 구현된 Top 10 → 자동 분석 / Top 5 → 자동 입력 흐름이 운영 환경에서도 재현 가능
4. 배포 후 최소 smoke test와 CI 검증 기준이 문서화되어 재배포가 쉬운 상태가 될 것

---

## 2. 현재 기준 상태

확인된 현재 상태:

- 프론트 로컬 기준 주소: `http://localhost:3000`
- 백엔드 로컬 기준 주소: `http://localhost:8000`
- 프론트 `test:ci` 존재
- Playwright E2E 존재
- GitHub Actions 프론트 CI 존재
- 백엔드 `/health` 엔드포인트 존재
- `docker-compose.yml`은 개발용에 가까움
  - backend: `uvicorn ... --reload`
  - frontend: `npm run dev`
  - 즉, 현재 compose는 운영 배포용이 아니라 로컬 개발용 기준

중요한 해석:

- 지금 바로 운영 배포를 한다면 **개발용 compose를 그대로 쓰면 안 됨**
- 운영 배포에서는 `--reload`, `npm run dev`, `localhost` 기준 env를 정리해야 함

## 3. 추천 배포 방식

### 권장 1안: 프론트/백엔드 분리 배포

가장 현실적인 1차 배포안:

- **Frontend:** Vercel 또는 Node 호스팅
- **Backend:** Railway / Render / Fly.io / VPS Docker

이 방식의 장점:

- Next.js와 FastAPI의 운영 특성을 분리해 관리 가능
- 프론트 재배포와 백엔드 재배포를 독립적으로 수행 가능
- 환경변수와 CORS 문제를 더 명확하게 분리 가능

### 권장 2안: Docker 기반 단일 서버 배포

선택지:

- Ubuntu VPS + Docker + reverse proxy(Nginx/Caddy)

이 방식의 장점:

- 전체 제어권이 높음
- 공모전 데모용으로 도메인/SSL/로그 접근을 직접 제어 가능

이 방식의 단점:

- 초기 운영 세팅 부담이 더 큼
- HTTPS, reverse proxy, process restart, log rotation까지 직접 챙겨야 함

### 이번 프로젝트 기준 추천

**가장 빠른 배포 목표라면 `프론트/백엔드 분리 배포`를 추천**한다.

- 프론트: Vercel
- 백엔드: Railway 또는 Render

이 조합이 가장 빨리 외부 데모 URL을 만들기 쉽다.

## 4. 간단 배포 단계 계획

### 단계 1: 배포 대상 환경 확정

결정할 것:

- 프론트 배포처: `Vercel` 권장
- 백엔드 배포처: `Railway` 또는 `Render` 권장
- 최종 공개 도메인/서브도메인 구조
  - 예: `app.example.com` (frontend)
  - 예: `api.example.com` (backend)

산출물:

- 배포 플랫폼 확정
- 프론트 URL / 백엔드 URL 초안

### 단계 2: 운영용 환경변수 정리

정리할 변수:

#### Frontend
- `NEXT_PUBLIC_API_BASE_URL`
- Kakao JavaScript key 관련 공개 변수

#### Backend
- `OPENAI_API_KEY`
- `WILDFIRE_API_KEY`
- `KAKAO_REST_API_KEY`
- `BACKEND_CORS_ORIGINS`
- 데이터 파일 경로 관련 변수

체크포인트:

- 프론트는 더 이상 `http://localhost:8000`을 쓰면 안 됨
- 백엔드 CORS는 실제 프론트 배포 URL 기준으로 바뀌어야 함

### 단계 3: 운영 빌드/실행 명령 확정

운영 기준으로 정리할 것:

#### Frontend
- build: `npm run build`
- start: `npm run start -- --hostname 0.0.0.0 --port 3000`

#### Backend
- run: `python3 -m uvicorn app.main:app --host 0.0.0.0 --port 8000`

체크포인트:

- `npm run dev` 사용 금지
- `uvicorn --reload` 사용 금지
- 플랫폼 기본 포트/런타임 요구사항 확인

### 단계 4: 배포 전 품질 게이트 수행

배포 직전 최소 검증:

#### Frontend
```bash
cd frontend
npm run test:ci
npm run build
```

#### Backend
```bash
cd backend
python3 -m uvicorn app.main:app --host 127.0.0.1 --port 8000
curl http://127.0.0.1:8000/health
```

가능하면 추가:
- 배포 전 실제 `.env` 기준 수동 분석 1회
- Top 10 → `/analyze` 자동 분석 흐름 브라우저 확인 1회

### 단계 5: 백엔드 먼저 배포

이유:

- 프론트는 API base URL에 의존하므로 백엔드 주소가 먼저 확정되어야 함

배포 후 확인:

- `/health` 정상 응답
- `/api/monitoring/summary` 응답
- `/api/analyze` 응답
- OpenAI unavailable 시 fallback도 최소한 실패 없이 응답하는지 확인

### 단계 6: 프론트 배포

백엔드 공개 URL 확정 후:

- `NEXT_PUBLIC_API_BASE_URL`을 운영 백엔드 주소로 설정
- 프론트 production build/deploy

배포 후 확인:

- 홈(`/`) 렌더링
- 모니터링 데이터 로드
- Top 10 카드 클릭 시 `/analyze?...&autoRun=1` 이동
- 자동 분석 요청 성공
- Top 5 링크는 자동 입력만 수행

### 단계 7: 배포 후 smoke test

실운영 URL에서 확인할 최소 체크리스트:

1. 홈페이지 진입 성공
2. monitoring summary 카드 로딩 성공
3. Top 10 카드 클릭 성공
4. analyze 자동 실행 성공
5. 결과 카드(위험도/XAI/오탐 요약) 렌더링 성공
6. Top 5 prefill-only 동작 성공
7. 브라우저 콘솔 치명 오류 없음

### 단계 8: 운영 문서/재배포 절차 정리

정리할 문서:

- 배포 URL
- 환경변수 목록(값 제외)
- 재배포 순서
- 장애 시 체크 포인트
- smoke test 순서

## 6. 리스크와 주의사항

### 리스크 1: CORS mismatch
가장 흔한 초기 장애 포인트다.

대응:
- 백엔드 `BACKEND_CORS_ORIGINS`에 실제 프론트 도메인 반영
- `localhost` 값만 남겨두지 않기

### 리스크 2: 프론트 API base URL 오설정
대응:
- 운영 배포 후 브라우저 network tab에서 실제 호출 URL 확인
- `NEXT_PUBLIC_API_BASE_URL`이 dev URL로 남아 있지 않은지 확인

### 리스크 3: OpenAI 또는 외부 API 키 문제
대응:
- 키 없는 경우 fallback 경로라도 동작하는지 확인
- 공모전 데모 시에는 live path / fallback path 둘 다 설명 가능하도록 준비

### 리스크 4: 개발용 실행 명령 배포
대응:
- `npm run dev`, `uvicorn --reload`를 운영에서 사용하지 않기
- production start 명령 분리 확인

### 리스크 5: 데이터 파일 경로 불일치
대응:
- 배포 환경에서도 필요한 processed data가 접근 가능한지 확인
- 파일 기반이면 이미지/데이터 볼륨 또는 repo 포함 전략 결정
