# 산불 의사결정 지원 웹 서비스 최종 보고서

## 1. 프로젝트 개요

### 프로젝트명
**에이전트 AI 기반 산불 예측·감지·의사결정 자동화 웹 서비스**

### 한 줄 소개
사용자가 지도에서 위치를 선택하면, 에이전트 AI가 **과거 산불 이력·경향성 데이터·실시간 산불 API**를 상황에 맞게 조회하고, **위험도·오탐 가능성·대응 가이드**를 함께 제시하는 웹 서비스이다.

### 해결하려는 문제
기존 산불 탐지/예측 시스템은 대체로 다음 한계를 가진다.
- 단일 모델 또는 단일 센서 결과에 의존
- 안개, 구름, 저시정 등으로 인한 **오탐(false positive)** 발생
- 사용자가 직접 해석해야 하는 확률 점수 위주 출력
- 현장 대응에 필요한 **즉시 행동 가능한 정보** 부족

본 프로젝트는 이 한계를 **Agentic AI(에이전트 AI)** 로 보완하는 데 초점을 둔다.

---

## 2. 왜 "에이전트 AI" 인가

이 서비스의 핵심 차별점은 단순히 LLM을 붙인 챗봇이 아니라, **상황에 따라 필요한 데이터 소스를 선택하고, 여러 검토 단계를 거쳐 응답을 조정하는 에이전트형 분석 구조**라는 점이다.

### 기존 AI 방식
- 입력 → 단일 모델 추론 → 결과 출력
- 데이터 조회 여부나 검증 절차가 고정적임

### 본 서비스의 에이전트 방식
- 입력 위치와 사용자 유형을 받음
- LLM/룰 기반 선택 레이어가 **무엇을 조회할지 결정**
  - 과거 이력 조회
  - 실시간 API 조회
  - 둘 다 조회
- LangGraph 기반 단계형 워크플로우 실행
- 1차 위험도 평가 후,
  - 오탐 검토
  - 데이터 품질 검토
  - 소스 검토
  - 점수 보정
- 최종적으로 **행동 가이드까지 포함한 결과** 제공

즉, 이 서비스는 “모델 1회 추론”이 아니라,
**조회 → 판단 → 검증 → 보정 → 설명**의 루프를 갖는 산불 대응용 에이전트 AI 서비스다.

## 3. 서비스 핵심 가치

### 3-1. 오탐 보정
산불로 오인하기 쉬운 조건을 별도 검토해, 단순 위험도와 별도로 **오탐 가능성(false positive risk)** 을 제시한다.

### 3-2. 맥락 기반 판단
기상, 과거 산불 이력, 위치 기반 맥락, 실시간 상태를 함께 반영해 더 실제적인 판단을 수행한다.

### 3-3. 즉시 행동 가능한 결과
결과를 단순 점수로 끝내지 않고,
- 위험 수준
- 주요 근거
- 불확실성 메모
- 추천 대응 행동
으로 구조화해 제공한다.

### 3-4. 사용자 유형별 대응
공무원, 진압대원, 구조요원, 시민 등 **사용자 유형**에 따라 실무적 활용성을 고려한 응답 설계를 갖는다.

## 4. 시스템 구성

## 4-1. 아키텍처
본 프로젝트는 **현대적 MVC Model2 스타일 split stack** 구조를 사용한다.

- **Frontend (View)**: Next.js + TypeScript
- **Backend (Controller + Domain)**: FastAPI + Python
- **Agent Workflow**: LangGraph
- **Data/Service Layer**: historical / realtime / risk / false-positive review

### 디렉터리 구조
- `frontend/` : 사용자 웹 UI
- `backend/app/api/routes/` : API 엔드포인트
- `backend/app/services/` : 데이터 조회, 정규화, 위험 분석
- `backend/app/agents/` : LangGraph 기반 에이전트 오케스트레이션
- `scripts/` : 데이터 전처리/ingest
- `docs/` : 문서 및 보고서

## 5. 에이전트 분석 흐름

현재 백엔드는 LangGraph 기반으로 다음 흐름을 가진다.

1. **select_sources**  
   - Group E 선택 레이어
   - LLM 또는 fallback 규칙이 historical / realtime 조회 여부 결정

2. **fetch_historical**  
   - 과거 산불 경향성 및 인근 이력 조회

3. **fetch_realtime**  
   - 실시간 산불 API 상태 조회

4. **derive_features**  
   - 분석용 feature 조립

5. **primary_assessment**  
   - 1차 위험도 계산

6. **verification_gate**  
   - 추가 검토 필요 여부 판단

7. **false_positive_review**  
   - 오탐 가능성 점검

8. **data_quality_review**  
   - 추정치/결측치/데이터 품질 영향 반영

9. **source_review**  
   - historical / realtime 소스 상태 검토

10. **decision_adjustment**  
   - 점수 및 설명 보정

11. **compose_report / build_response**  
   - 최종 사용자 응답 생성

이 구조 덕분에, 서비스는 단순 조회형이 아니라 **검토형 Agent Workflow** 로 동작한다.

## 6. 웹 사용 방법

## 6-1. 사용자 사용 흐름
1. 웹 페이지 접속
2. 지도에서 위치 클릭 또는 위도/경도 직접 입력
3. 사용자 유형 선택
   - 공무원
   - 진압대원
   - 구조요원
   - 시민
4. **분석 실행** 버튼 클릭
5. 결과 확인
   - 위험 수준
   - 위험 점수
   - 오탐 가능성
   - 주요 판단 근거
   - 추천 행동 가이드
   - 어떤 소스를 선택했는지(selection metadata)

## 6-2. 웹 UI 핵심 요소
- **지도 패널**: Kakao Map 기반 위치 선택
- **위치 입력 폼**: 위도/경도 수동 입력 가능
- **위험 요약 카드**: 위험 레벨 / 점수 / confidence
- **오탐 검토 패널**: 불확실성과 오탐 관련 메모
- **행동 가이드 카드**: 실제 대응 행동 제안
- **선택 메타데이터 카드**: 에이전트가 어떤 도구를 왜 골랐는지 설명

즉, 사용자는 단순히 “이 지역 위험합니다”를 보는 것이 아니라,
**왜 그렇게 판단했는지와 무엇을 해야 하는지**까지 확인할 수 있다.


### 7-4. `/api/analyze` 실호출 검증

- `risk_level`: 반환됨

즉, 현재 구현은 단순 규칙 응답이 아니라 **선택 레이어 + LangGraph 흐름이 실제 응답에 반영되는 상태**다.

### 7-5. 산림 API 유효성 검증
실제 점검 결과:
- API key 존재 확인
- 실시간 API base URL 설정 확인
- 실제 호출 시
  - `available=True`
  - `source=live`
  - `reason=None`
  - 네트워크 오류 없음

이 의미는 다음과 같다.
- **API 연결 자체는 유효**함
- 다만 검증 시점 기준으로 반환된 활성 산불 항목 수가 0건이었음
- 즉, **연동 실패가 아니라 정상 조회 결과가 빈 결과**인 상황으로 해석 가능

## 8. 기술적 차별성

본 프로젝트의 차별성은 아래 네 가지로 정리할 수 있다.

### 1) Agentic Source Selection
모든 요청에서 항상 같은 데이터를 기계적으로 보는 것이 아니라,
**LLM 선택 레이어가 historical / realtime 조회 여부를 결정**한다.

### 2) LangGraph 기반 검토 루프
단일 함수 호출이 아니라,
**위험 평가 → 오탐 검토 → 데이터 품질 검토 → 소스 검토 → 점수 보정** 구조를 가진다.

### 3) 웹 서비스형 구현
연구용 노트북이 아니라,
실제 사용 가능한 **지도 기반 웹 인터페이스**로 구현되어 있다.

### 4) 설명 가능한 결과
출력값이 단순 확률 점수에 그치지 않고,
선택 이유, 근거, 불확실성, 대응 행동을 함께 제시한다.

## 9. 한계와 개선 방향

### 현재 한계
1. 실시간 API의 위치 정보가 제한적일 수 있음
2. 일부 feature는 실시간 계측값이 아니라 추정치/가공치일 수 있음
3. 지도 위 과거 산불 포인트/히트맵 시각화는 추가 고도화 여지 있음
4. 사용자 유형별 응답을 더 세분화할 수 있음

### 향후 개선 방향
1. 실시간 기상 API 추가 연동
2. 위치 기반 과거 산불 시각화 강화
3. 지역별 맞춤 대응 시나리오 추천
4. 대시보드형 운영 모드 추가
5. 모바일 최적화 및 배포 고도화

## 10. 사용자가 추가로 제공하면 좋은 것

현재 로컬 개발 기준으로 핵심 요소는 이미 많이 갖춰져 있다.  
다만 **최종 배포/발표용 완성도**를 위해 사용자가 추가로 준비하면 좋은 것은 아래와 같다.

### 필수에 가까운 것
1. **배포 도메인 또는 최종 실행 URL**
   - Kakao Developers에 허용 도메인 등록 필요
2. **발표용 대표 시나리오 2~3개**
   - 예: 공무원 시나리오 / 시민 시나리오 / 진압대원 시나리오
3. **최종 제출용 서비스명/슬로건**
   - 보고서/발표자료 통일성을 위해 필요

### 있으면 좋은 것
4. **공모전용 스크린샷 또는 데모 캡처용 좌표 세트**
5. **심사위원 대상 설명 문구**
   - “왜 이게 agent AI인가?”를 짧게 전달할 문장
6. **최종 배포 환경 정보**
   - 로컬 데모인지, Vercel/Cloud Run/기타 배포인지

즉, 지금 시점에서 사용자가 꼭 새로 제공해야 하는 개발 요소는 많지 않고,
**배포/발표용 운영 정보와 데모 시나리오**가 가장 중요하다.

## 11. 결론

본 프로젝트는 단순 산불 탐지 웹페이지가 아니라,
**에이전트 AI가 상황에 맞는 데이터 조회를 선택하고, 검토 루프를 통해 위험도와 대응 가이드를 생성하는 산불 의사결정 지원 웹 서비스**다.

핵심 성과는 다음과 같다.
- 지도 기반 위치 입력 웹 UI 구현
- 과거 산불 데이터 + 실시간 API + 위험 분석 로직 통합
- Group E 기반 LLM/tool selection 레이어 반영
- LangGraph 기반 에이전트 워크플로우 반영
- 테스트 통과 및 빌드 검증 완료

따라서 이 서비스는 공모전 관점에서 다음 메시지를 강하게 전달할 수 있다.

> **"기존 AI가 단일 탐지 결과를 보여주는 수준에 머물렀다면, 본 프로젝트는 에이전트 AI가 상황에 맞는 정보를 선택적으로 조회하고 검토하여, 실제 의사결정 가능한 산불 대응 정보를 제공하는 웹 서비스이다."**
