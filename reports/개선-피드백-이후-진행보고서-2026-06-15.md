## 2. 개선 피드백 기준 정리

이번 단계에서 우선 반영 대상으로 잡은 개선 피드백은 다음과 같다.

1. 모니터링 카드 UX를 다듬고 우선순위 해석을 쉽게 만들 것
2. Top 10 항목 클릭 시 `/analyze`가 자동 실행되도록 만들 것
3. Top 5 항목은 자동 분석이 아니라 좌표/반경 자동 입력만 유지할 것
4. `/analyze` 진입 직후 로딩 상태를 더 명확하게 보여줄 것
5. 프론트 핵심 흐름을 수동 확인에만 의존하지 말고 자동화 검증까지 묶을 것
6. 가능하면 CI형 검증 흐름으로 연결할 것

또한 장기적으로는 다음 과제가 함께 존재한다.

- confidence margin 및 XAI 설명 강화
- 2페이지 운영 흐름 정교화
- 공간 반경 처리 검증 고도화
- 지도 전역 위험도 및 저신뢰 구간 재검토 흐름 확장
- 백엔드 테스트/배포 하드닝

## 3. 이번 단계에서 실제 반영된 주요 사항

### 3.1 모니터링 개요 화면 UX 개선

`frontend/components/monitoring-overview.tsx` 기준으로 모니터링 화면을 운영자 친화적으로 다듬었다.

주요 변경점:

- **관제 우선순위 Top 10** 영역에 우선순위 설명과 요약 스트립을 추가했다.
- 각 항목에 대해 누적 건수, 최근 연도, 주요 원인, 우선순위 점수를 한 번에 읽을 수 있도록 정리했다.
- Top 10 카드의 액션 문구를 유지하면서도 카드 자체를 더 직관적으로 탐색할 수 있게 구조를 정리했다.
- Top 5 영역은 “좌표 기반 현장 확인용”이라는 역할이 명확하게 드러나도록 문구와 칩 구성을 정리했다.

효과:

- 운영자가 홈 화면에서 우선 감시 지점과 빠른 진입 지점을 더 쉽게 구분할 수 있게 됨
- 우선순위 카드가 단순 링크 목록이 아니라 상황판 역할을 하도록 개선됨

### 3.2 Top 10 전체 카드 클릭 + `/analyze` 자동 실행 흐름 구현

Top 10 항목은 이제 단일 outer `Link` 기반의 **전체 카드 클릭 구조**로 동작한다.

적용 방식:

- Top 10 항목에 대해 `buildAnalyzeWatchpointHref(watchpoint, true)`를 사용해 `autoRun=1`이 포함된 analyze URL 생성
- 중첩 interactive element 문제를 피하기 위해 내부 CTA는 별도 링크가 아니라 안내 텍스트로 유지
- 클릭 시 `/analyze?lat=...&lon=...&radiusKm=...&source=...&autoRun=1` 형식으로 이동

효과:

- 사용자가 링크 텍스트만 클릭하지 않아도 카드 전체를 클릭해 분석으로 진입 가능
- “모니터링 상위 지점 → 즉시 분석” 흐름이 한 번의 클릭으로 단축됨

### 3.3 Top 5는 자동 입력만 유지

Top 5는 의도적으로 Top 10과 다르게 유지했다.

동작 원칙:

- 위도/경도/권장 반경/출처는 자동 입력
- `autoRun=1`은 붙이지 않음
- 사용자가 분석 페이지에서 직접 `분석 실행` 버튼을 눌러 정밀 검토 수행

이유:

- Top 5는 “현장 확인용 좌표 전달” 성격이 강하므로, 무조건적인 자동 분석보다 사용자의 최종 확인 단계를 남기는 것이 안전함
- 운영 시나리오를 Top 10(즉시 분석) / Top 5(좌표 전달)로 분리해 UX 목적이 더 명확해짐

### 3.4 `/analyze` 자동 실행 로딩 UX 및 skeleton 추가

`frontend/components/analysis-dashboard.tsx`에서 자동 진입과 수동 실행의 상태를 구분해 보여주도록 개선했다.

핵심 반영 내용:

- 실행 출처 상태 추가: `auto` / `manual`
- 분석 phase 상태 추가: `idle`, `prefill`, `loading`, `success`, `error`
- 자동 진입 시 상단 배너에서 “모니터링 페이지에서 전달된 위치를 기준으로 자동 분석을 시작했습니다” 메시지 표시
- 첫 자동 실행 시 결과 카드가 비어 보이지 않도록 skeleton/준비 카드 렌더링
- 이미 결과가 있는 상태에서 재실행될 경우, 기존 결과를 유지하면서 refresh 안내 배너 표시

효과:

- 사용자가 “클릭했는데 아무 일도 안 일어나는 것처럼 보이는 문제”를 겪지 않음
- 자동 분석과 수동 분석의 차이가 더 명확해짐
- 결과 도착 전 레이아웃 점프가 줄어듦

### 3.5 스타일 시스템 보강

`frontend/app/globals.css`에 이번 흐름을 지원하는 스타일을 추가했다.

반영된 대표 클래스:

- `.monitoring-list__item--link`
- `.state-banner--auto-run`
- `.statusCard--loading`
- `.card--skeleton`
- `.skeletonStack`
- `.skeletonLine`
- `.cards--loading`
- `.cards--refreshing`

효과:

- Top 10 카드의 hover/focus 가시성 향상
- 자동 분석 로딩 UX 일관성 확보
- Playwright 검증 시 사용자 관찰 포인트가 더 명확해짐

### 3.6 E2E 테스트 추가

Playwright 기반 E2E 테스트를 추가해 모니터링 → 분석 handoff를 자동 검증할 수 있게 했다.

추가 파일:

- `frontend/playwright.config.ts`
- `frontend/e2e/monitor-to-analyze.spec.ts`

검증 시나리오:

1. **Top 10 full-card click → autoRun**
   - 모니터링 응답 mock
   - Top 10 카드 클릭
   - `/analyze?...&autoRun=1` 확인
   - 로딩 배너 및 skeleton 표시 확인
   - `/api/analyze` 자동 요청 payload 확인
   - 결과 카드 렌더링 확인

2. **Top 5 prefill only**
   - Top 5 링크 클릭
   - 좌표/반경 자동 입력 확인
   - 자동 `/api/analyze` 요청이 발생하지 않음을 확인
   - 사용자가 직접 `분석 실행` 버튼을 눌렀을 때만 요청 발생 확인

의미:

- 핵심 사용자 흐름이 수동 브라우저 확인만으로 남지 않음
- 향후 회귀 버그를 잡을 수 있는 최소한의 브라우저 검증 안전망을 확보함

리포지토리 루트에 GitHub Actions 워크플로를 추가하여 프론트 핵심 흐름을 CI에서 자동 검증할 수 있게 했다.

추가 파일:

- `.github/workflows/frontend-playwright-ci.yml`

프론트 스크립트 보강:

- `frontend/package.json`
  - `test:unit`
  - `test:e2e`
  - `test:e2e:headed`
  - `test:ci`

`test:ci` 내용:

```bash
npm run test:unit && npm run test:e2e -- e2e/monitor-to-analyze.spec.ts
```

CI 워크플로 요약:

1. checkout
2. Node 22 설정
3. `npm ci`
4. `npx playwright install --with-deps chromium`
5. `npm run test:ci`
6. `playwright-report`, `test-results` 아티팩트 업로드

효과:

- 프론트의 핵심 handoff 흐름을 PR/push 단위로 자동 검증 가능
- 실패 시 Playwright HTML 리포트와 raw results를 통해 원인 확인 가능

## 7. OpenAI 크레딧/키 이슈와 현재 동작 방식

현재 시스템은 OpenAI 소스 선택이 불가능한 경우에도 전체 분석 흐름이 멈추지 않도록 설계되어 있다.

동작 방식:

- OpenAI key가 없거나
- 크레딧/한도 이슈가 있거나
- selector 호출이 실패하면

`rule_fallback` 모드로 전환하여 다음처럼 계속 분석한다.

- historical source 사용
- realtime source 사용
- selection mode를 `rule_fallback`으로 표시
- 프론트에서 “LLM 소스 선택은 건너뛰었지만 규칙 기반으로 계속 분석했다”는 메시지 표시 가능

의미:

- OpenAI live path를 완전히 검증한 것은 아니지만
- UX 개선, handoff 구현, 프론트 검증, fallback 기반 분석 흐름 검증은 계속 진행 가능함

## 8. 현재 진행률 및 상태 평가

현 시점 기준 프로젝트 전체 진행률은 **약 73% 수준**으로 보는 것이 타당하다.

해석:

- 프론트 핵심 사용자 흐름은 상당 부분 정리됨
- 모니터링 → 분석 handoff는 실제 사용 가능한 수준까지 올라옴
- 자동화 검증 및 CI 기반의 최소 품질 게이트도 확보함
- 그러나 아래 항목은 아직 후속 작업이 필요함

남은 핵심 과제:

1. 공간 반경 처리와 endpoint/radius 해석 고도화
2. 지도 전역 위험도 및 저신뢰 구간 재검토 워크플로
3. 백엔드 pytest 환경 안정화 및 테스트 범위 확대
4. 실제 OpenAI 연동 경로의 live 검증
5. 배포/운영 환경 기준 문서화와 하드닝

## 9. 한계 및 유의사항

1. **백엔드 pytest 환경은 아직 정리 중**
   - 현재 Hermes 실행 환경에서는 `pytest` 모듈 부재로 직접 실행에 제약이 있었음
   - 대신 health 확인과 프론트 중심 검증은 완료함

2. **OpenAI live path는 fallback 중심으로 검증됨**
   - 실제 크레딧 복구 후 selector 품질은 별도 검증이 필요함

3. **지도 전역 위험도 렌더링은 아직 본격 구현 전 단계**
   - 현재는 모니터링 상위 지점 및 좌표 기반 분석 흐름이 우선 확보된 상태임

---

## 10. 결론

개선 피드백 이후 이번 단계에서 가장 중요한 성과는 다음 세 가지로 요약할 수 있다.

1. **모니터링 화면에서 분석 화면으로 넘어가는 사용자 흐름이 자연스러워졌다.**
   - Top 10은 즉시 분석, Top 5는 자동 입력이라는 역할 분리가 명확해졌다.

2. **자동 분석의 체감 품질이 좋아졌다.**
   - 배너와 skeleton을 통해 사용자는 현재 시스템이 무엇을 하고 있는지 즉시 이해할 수 있다.

3. **검증이 수동 확인 수준을 넘어 자동화되었다.**
   - unit/E2E/CI 흐름이 확보되어 회귀 방지 기반이 생겼다.

즉, 현재 프로젝트는 단순 시제품 단계를 넘어 **운영 시나리오가 드러나는 데모 가능한 수준의 프론트 흐름**을 확보한 상태이며, 다음 단계에서는 공간 처리 고도화와 backend/live validation 강화가 핵심 과제가 된다.
