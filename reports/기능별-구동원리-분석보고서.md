# 분석 보고서

## 목적
현재 프로젝트의 주요 기능에 대해, 실제 코드 기준으로 **무엇을 입력으로 받아**, **어떤 데이터/LLM을 사용하며**, **어떤 처리 로직을 통해 결과를 만드는지**를 역추적하여 정리한다.

---

## 1. 전체 구조 개요

### 1-1. 사용자 입력
프론트엔드에서 사용자가 입력하는 값은 다음과 같다.

- 위도 `lat`
- 경도 `lon`
- 사용자 유형 `user_type`
- 분석 반경 `radius_km`

프론트는 이 값을 `/api/analyze`로 다음과 같은 형태로 전달한다.

```json
{
  "lat": ...,
  "lon": ...,
  "user_type": "...",
  "radius_km": ...
}
```

### 1-2. 백엔드 처리 흐름
백엔드는 LangGraph 기반 워크플로우로 분석을 수행한다.

1. `select_sources`
2. `fetch_historical`
3. `fetch_realtime`
4. `derive_features`
5. `primary_assessment`
6. `verification_gate`
7. `false_positive_review`
8. `data_quality_review`
9. `source_review`
10. `decision_adjustment`
11. `compose_report`
12. `build_response`

즉, 현재 시스템은 단순 점수 계산기가 아니라 다음 순서를 갖는 **검토형 분석 파이프라인**이다.

> 입력 → 조회 소스 선택 → 데이터 조회 → feature 생성 → 위험도 계산 → 오탐/품질 검토 → 점수 보정 → 설명 생성

## 2. 분석 반경 처리

### 2-1. 사용 데이터
분석 반경 기능은 다음 입력을 사용한다.

- 사용자 입력 `radius_km`
- 입력 좌표 `lat`, `lon`
- 과거 산불 위치 데이터

관련 코드:

- `frontend/components/analysis-dashboard.tsx`
- `backend/app/schemas/report.py`
- `backend/app/services/group_b_context.py`
- `backend/app/services/spatial_radius.py`
- `backend/app/services/historical_wildfire.py`

### 2-2. 처리 방식

#### 프론트 입력
프론트에서는 사용자가 반경을 직접 입력하거나 빠른 선택 버튼으로 다음 값을 고를 수 있다.

- 1km
- 3km
- 5km
- 10km
- 20km

#### 백엔드 전달
선택된 반경은 `AnalyzeRequest.radius_km`로 전달된다.

#### 기본 반경
반경 입력이 없으면 기본값은 다음 상수를 사용한다.

```python
DEFAULT_HISTORICAL_ANALYSIS_RADIUS_KM = 50.0
```

즉 기본 반경은 **50km**이다.

#### 실제 사용처
현재 반경은 주로 **과거 산불 이력 조회 범위**를 결정하는 데 사용된다.

백엔드는 trend CSV에서 입력 좌표 주변 incident를 검색할 때 `radius_km` 이내만 선택한다.

```python
find_nearby_incidents(... radius_km=effective_radius_km)
```

거리 계산은 haversine 공식을 사용한다.

#### 지도 시각화
반경은 결과 응답에서 `radius_points`로도 변환된다.

- 북쪽
- 남쪽
- 동쪽
- 서쪽

즉, 분석 반경은 단지 숫자로 끝나는 것이 아니라 지도 원형 오버레이와 경계 좌표 계산에도 사용된다.

### 2-3. 정리
현재 분석 반경은 **전체 분석 논리를 직접 바꾸는 변수라기보다**, 과거 산불 이력 조회 범위를 정하는 핵심 파라미터다.

### 2-4. 사용 데이터/LLM 정리
- 데이터: 사용자 입력 반경, 과거 산불 위치 데이터
- 처리 로직: haversine 거리 계산, 위경도 경계 계산
- LLM 사용 여부: **없음**

## 3. 사용자 유형 구분 이유

### 3-1. 사용 데이터
입력 데이터는 `user_type`이다.

프론트에서 제공하는 선택지는 다음과 같다.

- 공무원
- 진압대원
- 구조요원
- 시민

### 3-2. 현재 코드에서의 역할

#### 조회 소스 선택 입력
`group_e_selector.py`에서 LLM 프롬프트에 `user_type`이 포함된다.
즉, LLM은 다음 정보를 함께 보고 조회 전략을 결정한다.

- lat
- lon
- user_type

#### 권장 행동 문구 분기
`wildfire_graph.py`의 `_build_actions()`에서 사용자 유형이 사용된다.

현재 구현상 `시민`은 별도 행동 가이드를 가지며, 나머지 유형은 기본 액션 세트를 사용한다.

### 3-3. 해석
현재 구현 기준으로 사용자 유형은 **위험 점수 자체를 바꾸지 않는다.**
대신 다음에 영향을 준다.

1. LLM 조회 소스 선택 입력값
2. 최종 추천 행동 문구

즉 사용자 유형 구분의 목적은 다음과 같이 해석할 수 있다.

> 같은 위험 상황이어도, 사용자 역할에 따라 필요한 설명과 권장 행동이 다르기 때문이다.

다만 현재 완성도 기준으로는:

- 프론트 선택지는 4개
- 백엔드의 실제 세부 분기는 `시민`과 `그 외` 수준

이므로 향후 고도화 여지가 있다.

### 3-4. 사용 데이터/LLM 정리
- 데이터: 사용자 유형 문자열
- 처리 로직: 행동 가이드 분기 + LLM 프롬프트 입력
- LLM 사용 여부: **조회 소스 선택 단계에서 간접 사용**

## 4. 위험도 요약 도출 과정

위험도 요약은 다음 요소로 구성된다.

- 위험 점수 `risk_score`
- 위험 단계 `risk_level`
- 판독 신뢰도 `confidence`
- 요약 문장 `risk_summary_text`

### 4-1. 위험 점수 계산 방식
관련 코드:

- `backend/app/services/risk_engine.py`

위험 점수는 규칙 기반으로 계산된다. 입력 feature는 다음과 같다.

- `humidity_percent`
- `wind_speed`
- `drought_duration_days`
- `slope`
- `fuel_moisture`
- `vulnerable`

가중치 규칙은 다음과 같다.

- 습도 30 미만: `+0.2`
- 풍속 5 이상: `+0.2`
- 가뭄 20일 이상: `+0.15`
- 경사 15 이상: `+0.1`
- 연료 수분 12 미만: `+0.15`
- 취약계층 지표 1000 이상: `+0.1`

최대 점수는 `0.99`로 제한된다.

위험 단계 구간:

- `>= 0.75`: `critical`
- `>= 0.5`: `high`
- `>= 0.25`: `medium`
- 그 미만: `low`

즉, 현재 위험도는 **LLM 추론이 아니라 규칙 기반 점수 합산 방식**이다.

### 4-2. 위험 점수에 쓰이는 feature 출처
관련 코드:

- `backend/app/services/group_b_context.py`

#### 과거 데이터 기반
- `drought_duration_days`
- `vulnerable`
- nearby incidents 평균 기반 기상 추정치
- fire intensity 추정 보조

#### 실시간 데이터 기반
- `humidity_percent`
- `wind_speed`
- `visibility`
- `surface_temperature`
- `fire_intensity`

#### 결측 시 대체 순서
현재 시스템은 다음 우선순위를 사용한다.

1. 실시간 값
2. 과거 nearby incident 평균값
3. 기본 default 값

예시 default 값:

- 습도: `45`
- 풍속: `1`
- 가시성: `5000`
- 표면온도: `24`

추정값을 사용한 항목은 `estimated_fields`에 기록한다.

### 4-3. 판독 신뢰도 계산 방식
관련 코드:

- `wildfire_graph.py` 내부 `_calculate_confidence()`

기본 신뢰도는 오탐 위험도에 따라 다르게 시작한다.

- false positive low → `0.84`
- false positive medium → `0.74`
- false positive high → `0.62`

이후 감점/가점 규칙:

- historical source가 `estimated`: `-0.05`
- historical source가 `fallback`: `-0.08`
- realtime source가 `live` 아님: `-0.07`
- estimated field 수에 따라 최대 `-0.10`
- 위험도 보정이 일어났으면: `-0.04`
- historical/realtime 둘 다 `live`이고 추정치 없으면: `+0.03`

최종 범위는 `0.35 ~ 0.93`으로 제한된다.

즉 현재 confidence는 “모델 확률”보다도 **데이터 품질과 보정 필요성에 기반한 운영형 신뢰도 지표**에 가깝다.

### 4-4. 위험도 요약 문장 생성 방식
관련 코드:

- `_build_risk_summary_text()`

이 문장은 템플릿 방식으로 생성된다.

구성 요소:

- 위험 단계
- 위험 점수
- 반경 내 과거 incident 개수

예시 구조:

> 현재 위치는 반경 Xkm 기준으로 산불 위험이 높은 상태입니다 (위험 점수 0.xx, 인접 이력 n건 참고).

즉 이 문장도 LLM 문장 생성 결과가 아니라 **문자열 템플릿 조합**이다.

### 4-5. 사용 데이터/LLM 정리
- 데이터:
  - 과거 CSV 인접 incident
  - 실시간 API 상태
  - 추정 default 값
- 처리 로직:
  - 규칙 기반 risk scoring
  - 규칙 기반 confidence scoring
  - 템플릿 기반 summary 생성
- LLM 사용 여부: **없음**

## 5. 조회 소스 선택 기준

대상:

- historical 조회 여부
- realtime 조회 여부
- 둘 다 조회할지 여부

### 5-1. 사용 데이터
관련 코드:

- `backend/app/services/group_e_selector.py`

입력값:

- `lat`
- `lon`
- `user_type`

### 5-2. LLM 사용 방식
현재 LLM은 OpenAI Responses API를 통해 호출된다.

```python
client.responses.create(...)
```

기본 모델 설정값:

- `gpt-5.4`

프롬프트 구조:

#### system prompt
- wildfire decision-support source selector 역할 지정
- historical / realtime / both 중 선택 요구
- JSON only 반환 요구

#### user prompt
- `lat`
- `lon`
- `user_type`
- available tools: historical, realtime
- mode='llm'

즉 이 단계의 LLM 역할은 다음과 같다.

> 이 좌표와 사용자 유형에서 과거 데이터, 실시간 데이터, 혹은 둘 다 조회해야 하는지 판단하는 도구 선택기

### 5-3. fallback 규칙
다음 경우에는 LLM을 사용하지 않는다.

- Group E 기능 비활성화
- OpenAI API key 없음
- provider가 openai가 아님
- 호출 실패, quota 문제, 429 등

이 경우 fallback은 다음과 같다.

- historical = True
- realtime = True
- 둘 다 조회

즉 fallback은 **보수적으로 모든 소스를 조회**하는 전략이다.

### 5-4. 해석
현재 LLM은 **위험도를 계산하지 않는다.**
LLM은 오직 다음 역할만 수행한다.

- 어떤 데이터를 읽을지 결정
- 그 이유를 설명 텍스트로 반환

### 5-5. 사용 데이터/LLM 정리
- 데이터: `lat`, `lon`, `user_type`
- LLM: OpenAI Responses API, `gpt-5.4`
- 출력:
  - `use_historical`
  - `use_realtime`
  - `selected_tools`
  - `reason`
  - `mode`

## 6. 분석범위의 신뢰도 오차 폭

대상:

- `confidence_margin`
- 화면의 “신뢰도 오차폭”

관련 코드:

- `backend/app/services/confidence_metrics.py`

### 6-1. 계산 방식
기본 margin:

- `0.11`

조정 규칙은 다음과 같다.

#### historical source
- `live`: `-0.02`
- `estimated`: `+0.02`
- `fallback`: `+0.03`

#### realtime source
- `live`: `-0.02`
- `fallback`: `+0.02`
- `skipped`: `+0.01`

#### nearby incident count
- 8건 이상: `-0.01`
- 1건 이하: `+0.01`

#### estimated fields
- 3개 이상: `+0.01`

최종 범위는 `0.05 ~ 0.20`으로 제한된다.

### 6-2. 의미
이 값은 엄밀한 통계적 신뢰구간이라기보다,

> 현재 데이터 품질과 추정치 비중을 고려했을 때, 결과 해석이 어느 정도 흔들릴 수 있는가

를 나타내는 운영형 불확실성 지표다.

### 6-3. 사용 데이터/LLM 정리
- 데이터:
  - historical source 상태
  - realtime source 상태
  - nearby incident 수
  - estimated field 개수
- 처리 로직: 규칙 기반 margin 계산
- LLM 사용 여부: **없음**

## 7. 사용자용 판단 근거 작성 과정

대상:

- 주요 근거 `key_factors`
- XAI 설명 `xai_reasons`
- 검토 신호 `reviewed_signals`
- uncertainty notes

관련 코드:

- `wildfire_graph.py`
  - `_build_key_factors`
  - `_build_xai_reasons`
  - `_build_reviewed_signals`
  - `_build_uncertainty_notes`

### 7-1. 주요 근거 생성
재료:

- 초기 risk engine의 `key_factors`
- 위험도 보정 발생 여부
- 오탐 재검토 필요 여부
- estimated field 존재 여부
- 실시간 데이터 신뢰도 제한 여부

즉 주요 근거는 다음을 합쳐서 만든다.

> 왜 위험 점수가 올라갔는지 + 왜 보수적으로 해석해야 하는지

### 7-2. XAI 설명 생성
재료:

- initial risk factors
- 최종 risk level / risk score
- false positive risk
- 주변 과거 산불 이력 건수
- 실시간 source가 live인지 여부
- estimated fields 목록

즉 XAI 설명은 다음을 구조적으로 보여준다.

- 어떤 요인 때문에 위험 판단을 했는지
- 최종 점수와 단계가 무엇인지
- 왜 신뢰도나 해석에 제한이 있는지

### 7-3. 검토 신호 생성
재료:

- 반경 내 incident 개수
- historical source
- realtime source
- 실시간 습도
- 실시간 풍속
- estimated field 개수

즉 검토 신호는 다음을 사용자에게 보여준다.

> 이번 분석에서 실제로 어떤 입력 신호를 확인했는가

### 7-4. uncertainty notes 생성
재료:

- false positive notes
- agent trace
- historical/realtime source 상태
- context key 목록
- estimated fields
- 입력 좌표

이 항목은 사용자 설명 요소이기도 하지만, 성격상 **디버깅/추적 로그**에 더 가깝다.

### 7-5. 핵심 결론
현재 사용자용 판단 근거 작성은 **LLM 생성형 설명이 아니라 규칙+템플릿 조합 방식**이다.

### 7-6. 사용 데이터/LLM 정리
- 데이터:
  - risk engine 결과
  - false positive 결과
  - historical/realtime source 상태
  - estimated fields
  - LangGraph trace
- 처리 로직: 템플릿 조합
- LLM 사용 여부: **없음**

## 8. LLM 사용처와 사용 방식

### 8-1. 현재 실제 사용처
현재 `/api/analyze` 분석 흐름에서 LLM이 실제로 쓰이는 곳은 사실상 **1개 지점**이다.

#### 사용처
- Group E source selector

#### 목적
- historical 조회 여부 판단
- realtime 조회 여부 판단
- 둘 다 조회할지 판단

#### 입력
- `lat`
- `lon`
- `user_type`

#### 출력
JSON:

- `use_historical`
- `use_realtime`
- `selected_tools`
- `reason`
- `mode`

### 8-2. 현재 LLM이 쓰이지 않는 영역
다음 항목은 현재 모두 규칙 기반이다.

- 위험 점수 계산
- 위험 등급 판정
- 오탐 위험 계산
- confidence 계산
- confidence margin 계산
- XAI 설명 문장
- 추천 행동 문구
- 분석 반경 계산

### 8-3. 해석
현재 프로젝트는 다음 구조에 가깝다.

> LLM이 전체 분석을 수행하는 시스템이 아니라,
> LLM은 조회 전략을 선택하고,
> 나머지는 규칙 기반 분석 엔진이 처리하는 하이브리드 구조

## 10. 종합 결론

### 10-1. 구조적 결론
현재 프로젝트의 분석 엔진은 **완전한 LLM 분석기**가 아니다.

구조는 다음과 같다.

- LLM: 조회 소스 선택
- 규칙 엔진: 위험도, 오탐, 신뢰도, 설명 생성

### 10-2. 사용자 유형 반영 수준
사용자 유형은 존재하지만 현재는 다음 수준으로 반영된다.

- 조회 소스 선택 프롬프트 입력
- 추천 행동 문구 분기

즉, 아직은 `시민 vs 나머지`에 가까운 부분 구현 상태이며 향후 세분화 가능성이 크다.

### 10-3. 반경의 의미
분석 반경은 현재 위험 계산 전체를 뒤흔드는 파라미터라기보다,
**과거 산불 이력 검색 범위**를 정하는 핵심 입력값이다.

### 10-4. confidence 계열의 의미
`confidence`와 `confidence_margin`은 통계 모델 출력이라기보다,
**입력 데이터 품질과 보정 필요성을 반영한 운영형 해석 지표**에 가깝다.

### 10-5. 설명 생성 방식
현재 XAI, 요약 문장, 데이터 품질 문구, 권장 행동 등은 대부분 **LLM 생성이 아니라 템플릿 조합**으로 만들어진다.

## 11. 한 줄 요약

이 프로젝트의 현재 분석 엔진은,
**LLM이 좌표와 사용자 유형을 보고 어떤 데이터 소스를 조회할지 먼저 선택하고, 이후에는 과거 산불 CSV와 실시간 API 데이터를 바탕으로 규칙 기반 위험도·오탐·신뢰도·설명 문구를 계산하는 하이브리드 구조**이다.
